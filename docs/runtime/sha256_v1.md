# SHA-256 + measured boot — contract v1

Status: boot-verified via `make test-sha256-v1`
Source: `kernel_rs/src/sha256.rs` (`sha256`, `pcr_extend`, `sha256_selftest`),
`kernel_rs/src/aes.rs` (`sbox_bytes`).
Proof: `tests/runtime/test_sha256_v1.py`.

Full-OS guide Part IV.10 (security), secure boot: the integrity primitive
(SHA-256) and a TPM-style measured boot built on it.

## Behaviour

A from-scratch `no_std` SHA-256 (FIPS 180-4): the message schedule, compression
function, and `0x80`/zero/64-bit-length padding over 64-byte blocks (handling the
two-block final case). `pcr_extend(pcr, data)` folds a component into a PCR-like
accumulator: `pcr = SHA-256(pcr || SHA-256(data))` — exactly a TPM PCR extend
(extend with the component's digest).

`sha256_selftest` verifies the hash against three known-answer vectors (the empty
string, `"abc"`, and a 56-byte message that forces a second padded block), then
performs a **measured boot**: it extends a zeroed PCR with two real kernel
components (the SHA-256 round constants and the AES S-box) and records the
resulting measurement.

## Acceptance

`make test-sha256-v1`: the boot transcript shows `SHA256: kat ok` (all three KAT
vectors match) and `MEASURE: pcr=0x<...>` (the measured-boot PCR after extending
with two components, non-zero), with no `SHA256: kat fail`.

## v1 boundary / carry-forward

- SHA-256 + PCR-extend measurement. The **signature-verification** half of secure
  boot — checking the measurement/image against a trusted public key before
  execution (RSA/ECDSA + a key store, and a real TPM/PCR interface) — is
  carry-forward. This slice provides the hash those build on.
- One-shot `sha256(&[u8])` (no streaming `update`/`finalize` API yet).
