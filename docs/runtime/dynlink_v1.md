Full-OS implementation guide Part V.11 (userspace), dynamic loading: load
separately-authored code at runtime, resolve a symbol, and call it
(`dlopen`/`dlsym` semantics).

## A real ELF `.so` linker

v1 is a genuine ELF64 dynamic linker: it parses program headers, maps the
`PT_LOAD` segments, and **applies dynamic relocations** before handing back
symbol addresses. It does *not* depend on the broken C `.so` toolchain — the
shared object is authored in assembly (`apps/dl/libdl.asm`) and linked as a real
position-independent ELF shared object with `nasm -f elf64` + `rust-lld -shared`
(the mingw refptr/auto-import + `tools/pe_to_elf_v1.py` path could not produce a
working multi-page C `.so`; assembly + `rust-lld -shared` sidesteps it entirely).
The kernel `include_bytes!`s that `.so` and links it at load time.

The embedded `libdl` exports four functions, each gated on a different relocation:

- `addtwo(x)` = `lea rax,[rdi+2]; ret` — position-independent, no relocation;
- `getval()` reads a global through a pointer `pmyval` that holds the absolute
  address of `myval` (`= 42`). That pointer is an **`R_X86_64_RELATIVE`**
  relocation: it is wrong until the loader fixes it up, so `getval() == 42` is
  direct proof the relocation was applied;
- `getgvar()` reads the global `gvar` (`= 99`, defined in `libdl2`) **through the
  GOT** — an **`R_X86_64_GLOB_DAT`** symbolic relocation (the GOT slot must be
  resolved from `.dynsym` to `&gvar`), so `getgvar() == 99` proves GLOB_DAT;
- `callsum()` calls `extadd(40)` (`= 42`, defined in `libdl2`) **through the PLT**
  — an **`R_X86_64_JUMP_SLOT`** symbolic relocation (the `.got.plt` slot must be
  eager-bound to `&extadd`), so `callsum() == 42` proves JUMP_SLOT.

## Behaviour

`sys_dlctl(op, name_ptr)`:

- **op 1 = dlopen(name):** for the embedded object (`name == "libdl"`),
  `dl_load_elf()`:
  1. validates the ELF64 header (magic `\x7FELF`, `e_ident[4] == 2`);
  2. **pass 1** — for each `PT_LOAD`, maps `[DLOPEN_BASE+p_vaddr,
     +p_memsz)` page-by-page **RW** (`vm_map_current` then forced
     `vm_protect_current(RW)` so a repeat dlopen can re-copy) and `copyout_user`s
     the `p_filesz` file bytes in (bounds-checked against the image);
  3. **pass 2** — walks `PT_DYNAMIC` for `DT_RELA`/`DT_RELASZ` and applies each
     entry: `R_X86_64_RELATIVE` (`*(base+off) = base+addend`) and the **symbolic**
     `R_X86_64_GLOB_DAT` (`*(base+off) = base + sym.st_value + addend`, the symbol
     resolved from `.dynsym` by `r_info`'s symbol index via `dl_sym_value`);
  4. **pass 2b** — walks `DT_JMPREL`/`DT_PLTRELSZ` (the `.rela.plt`) and applies
     each **`R_X86_64_JUMP_SLOT`** with the same symbolic formula — **eager**
     binding, since the kernel has no lazy PLT resolver, so every PLT slot is
     bound at load. Each store is bounds-checked via `copyout_user`;
  5. **pass 3** — re-protects each `PT_LOAD` to its final perms from `p_flags`
     (`X → R-X`, `W → RW`, else `R`), enforcing **W^X** (a segment is never
     writable and executable at once).

  Returns `DLOPEN_BASE`. `DLOPEN_BASE = 0x0180_0000` sits in the one genuinely
  free 1 MiB sub-gap of the user demand window: that window
  `[0x0100_0000, 0x0200_0000)` is otherwise fully partitioned — brk
  `[0x100,0x120)`, mmap `[0x120,0x140)`, exec-app `[0x140,0x180)`, demand-task
  stacks `[0x190,0x200)` (MiB) — leaving only `[0x0180_0000, 0x0190_0000)`. The
  loaded object (shipped `.so` spans `<0x4000`) therefore aliases neither the
  caller's heap/mmap/exec segments nor a clone-thread's demand stack (which
  starts at `0x0190_0000`). Idempotent: a repeat dlopen re-maps/re-copies and
  returns the same base.
- **op 2 = dlsym(name):** `dl_resolve` finds `name` in the object's
  `.dynsym`/`.dynstr` (located via `DT_SYMTAB`/`DT_STRTAB`; the symbol count is
  the SysV `.hash` `nchain` word) and returns `DLOPEN_BASE + st_value`. Matching
  is exact (the `.so` name must terminate with `\0` right after the query), and
  every table access is bounds-checked against the image.

`dlprobe` resolves and calls all four exports: `getval()==42` (RELATIVE),
`addtwo(40)==42` (resolve + call), `getgvar()==99` (GLOB_DAT via the GOT), and
`callsum()==42` (JUMP_SLOT via the PLT) — each in ring 3.

## v1 boundary / carry-forward

- **One embedded `.so`, one load slot; RELATIVE + GLOB_DAT + JUMP_SLOT.** The
  loader applies relative *and* symbolic (GOT/PLT) relocations, eager-bound. What
  remains: **lazy** PLT binding (a runtime resolver stub), `DT_NEEDED` dependency
  chains, multiple simultaneously-loaded objects, and `dlclose` — carry-forward.
- **No on-disk/package `.so` files.** The object is kernel-embedded; loading a
  `.so` from the filesystem (and authoring C `.so` files) waits on the C `.so`
  toolchain fix — the documented blocker, now routed around for the assembly
  object.
- **Single fixed load address.** A relocating loader that places objects at an
  allocator-chosen base (and a second slot) is carry-forward.

## Acceptance

`make test-dynlink-v1`: the go lane runs `probe dlprobe`; the transcript shows
`DLPROBE: dlsym ok` — dlopen mapped the `.so` and applied RELATIVE + GLOB_DAT +
JUMP_SLOT relocations, dlsym resolved all four symbols from `.dynsym`, and the app
called each in ring 3 (`getval()==42` RELATIVE, `addtwo(40)==42` resolve+call,
`getgvar()==99` GLOB_DAT through the GOT, `callsum()==42` JUMP_SLOT through the
PLT) — with no `DLPROBE: FAIL` and no `USERPF`, then reaches `GOINIT: result
shutdown-clean` and `RUGO: halt ok`.

To regenerate the embedded object after editing `apps/dl/libdl.asm` or
`apps/dl/libdl2.asm`: `make dl-module` (writes `kernel_rs/src/dl_module.so`), then
rebuild and commit the new `.so`.
