# Intel e1000 NIC detection — contract v1

Status: boot-verified via `make test-e1000-v1`
Source: `kernel_rs/src/lib.rs` (`e1000_detect`, `e1000_report`,
`e1000_eeprom_read`; ATTACH entry in `pci_enumerate_log`).
Proof: `tests/runtime/test_e1000_v1.py`.

Full-OS guide Part II.7 (driver model + buses), the named second NIC (e1000):
discover the controller, map its registers, and read its identity. Mirrors the
xHCI detection slice ([`xhci_v1.md`](xhci_v1.md)).

## Behaviour

`e1000_detect` scans PCI bus 0 for vendor `0x8086` device `0x100E` (the QEMU
82540EM, `-device e1000`). On a match `e1000_report`:

1. enables memory space + bus-master (PCI command bits 0x0006);
2. reads BAR0 (a 32-bit memory BAR) and maps its page into the kernel MMIO window
   (`mmio_map_4k` — the PCI MMIO hole is not in the HHDM);
3. reads the **STATUS** register (offset 0x0008) — a present, decoding device
   returns a sane value, not the all-ones an unmapped BAR yields;
4. reads the **MAC** from the EEPROM via the EERD register (offset 0x14):
   `e1000_eeprom_read` writes `(word<<8)|START`, polls the DONE bit, and returns
   the data half; MAC = EEPROM words 0..2 (each holding two little-endian MAC
   bytes), packed into a 48-bit value.

The driver registry (`pci_enumerate_log`) also emits `ATTACH: e1000`. When no
e1000 is present (the default lane) it reports `E1000: none`.

## Acceptance

`make test-e1000-v1`: booting the go lane with `-device e1000,mac=52:54:00:12:34:56`,
the transcript shows `PCI: enumerate bus0`, `ATTACH: e1000`, and
`E1000: found status=0x<status> mac=0x0000563412005452` (the pinned MAC read back
out of the EEPROM), then a clean shutdown — with no `E1000: none`, no `E1000: bar
…`, no `E1000: eeprom timeout`, and no `E1000: mmio map fail`.

## TX ring driver (`e1000_tx_selftest`)

On detection the kernel also drives a real **transmit** path: it maps the first
16 KiB of the BAR (`mmio_map_region`, covering CTRL/TCTL @0x400 and the TX-ring
registers @0x3800), allocates a TX descriptor ring + a packet buffer from the DMA
pool ([`dma_v1.md`](dma_v1.md)), sets link-up (CTRL.SLU), programs
TDBAL/TDBAH/TDLEN/TDH/TDT + TIPG + TCTL (EN|PSP|CT|COLD), builds a padded 60-byte
Ethernet frame, posts descriptor 0 (buffer addr + length + CMD `EOP|IFCS|RS`),
bumps TDT, and polls the descriptor's **Done** bit — which the device sets after
DMA'ing the frame out. Marker `E1000: tx ok`.

## RX ring driver (`e1000_rx_selftest`)

After TX, the kernel drives a real **receive** path with a genuine wire
round-trip. It allocates an RX descriptor ring (8 descriptors) + 8 receive buffers
from the DMA pool, programs RDBAL/RDBAH/RDLEN + RDH=0/RDT=7, and enables the
receiver via **RCTL** = EN | UPE (accept-all-unicast) | BAM (accept-broadcast) |
SECRC (strip CRC). UPE lets the unicast reply through without programming the
receive-address filter (RAL/RAH @0x5400 lie beyond the mapped 16 KiB BAR window).

It then transmits an **ARP request** ("who has 10.0.2.2, tell 10.0.2.15") out the
NIC. QEMU's user-mode (slirp) backend always answers ARP for its gateway
(10.0.2.2), so the **reply is delivered back into the RX ring** — a real inbound
frame, not a self-test loopback (the legacy e1000 in QEMU implements no
PHY/MAC loopback). The kernel polls RX descriptor 0's **Done** bit (periodically
reading the RDH register — an MMIO access / VM exit — so the backend gets a chance
to inject the reply), then verifies the received bytes are an ARP **reply**
(ethertype 0x0806, opcode 2) from **10.0.2.2**. Marker `E1000: rx ok len=0x<n>`.

## v1 boundary / carry-forward

- **Detection + EEPROM MAC + TX and RX descriptor rings (real wire round-trip).**
  MSI-X interrupt-driven completion (vs the polled Done bit), a continuously
  serviced RX ring with refill, and making the e1000 the *active* NIC (vs
  virtio-net) carrying the live IP stack are carry-forward.
- RX validation relies on the always-present slirp ARP responder; a NIC on a
  bridge/tap backend would need a peer to answer.
- Single controller (first match wins).
